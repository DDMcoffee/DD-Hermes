# DD Hermes 恒定锚点策略

## 结论

DD Hermes 不再依赖“每轮重新发明一批 agent”来维持目标和质量。

从这一轮开始，仓库固定两类恒定锚点：

- `Product Anchor`
  - 默认映射到 `Supervisor`
  - 负责把模糊业务意图压成可执行任务，并持续校准 `product_goal / user_value / non_goals / drift_risk`
- `Quality Anchor`
  - 默认映射到 `Skeptic`
  - 负责架构一致性、命名规范、错误处理、性能隐患、安全漏洞和证据缺口的最后审查

它们是**协议层常驻席位**，不是两个必须长期在线的子 agent。

## 为什么这样做

- 用户关心的是“目标别漂、质量别自证”，不是“名字上有几个 agent”。
- 常驻席位比常驻子 agent 更省资源，也更不容易把人格连续性误当成目标连续性。
- 单线程默认模型下，真正稳定的不是聊天上下文，而是 `contract + state + context + gate` 四层工件。

## 落盘位置

### 1. Contract

每个新任务都要声明：

- `product_goal`
- `user_value`
- `non_goals`
- `product_acceptance`
- `drift_risk`

### 2. State

每个新任务都要在 `workspace/state/<task_id>/state.json` 里落盘：

- `team.product_anchors`
- `team.quality_anchors`
- `team.anchor_policy`
- `product.*`
- `quality.*`

### 3. Dispatch / Context / Gate

- `dispatch-create` 必须暴露两个锚点的 assignment 真相
- `context-build` 必须把产品目标状态和质量审查状态打进 context
- `thread-switch-gate` 必须阻止“目标缺失或已漂移”的任务进入实现
- `quality-gate` 必须阻止“代码改了但质量锚点未审过”的任务宣称完成

## 角色分工

### Product Anchor

- 关注：
  - 这轮到底服务哪个用户价值
  - 这轮明确不做什么
  - 当前实现有没有开始跑偏
- 输出：
  - 清晰的 `product_goal`
  - 可核对的 `product_acceptance`
  - 漂移风险和阻塞条件

### Quality Anchor

- 关注：
  - 架构一致性
  - 命名规范
  - 错误处理完整性
  - 性能隐患
  - 安全漏洞
  - 证据链是否足够
- 输出：
  - 审查结论
  - 关键发现
  - 改进建议
  - 参考范例

## 降级规则

- 如果 `Quality Anchor` 与 `Executor` 或 `Supervisor` 重叠，必须标记 `degraded`。
- 如果 `Product Anchor` 没有明确 `product_goal`，任务不得进入实现态。
- 如果 `goal_drift_flags` 非空，优先回到当前线程做重校准，而不是继续扩大实现面。

## 本轮主线

`dd-hermes-anchor-governance-v1` 的完成标准是：

- 新任务模板默认带有产品锚点与质量锚点字段
- `state / context / dispatch / gate / schema` 都能识别这两个恒定锚点
- 相关 smoke test 覆盖这些新约束
