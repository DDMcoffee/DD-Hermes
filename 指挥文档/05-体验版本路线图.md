# DD Hermes 体验版本路线图

## 结论先行

当前最接近用户可体验版本的，不是“再补更多脚手架”，而是一个单任务可重复闭环：

1. `sprint-init`
2. `state-read / state-update`
3. `context-build`
4. `dispatch-create`
5. execution worktree 实现与验证
6. `git-commit-task`
7. `verify-loop / check-artifact-schemas`
8. `git-integrate-task`
9. `worktree-remove`
10. `archive / closeout`

这条链如果在 `main` 上稳定、可追溯、可重复执行，就构成 DD Hermes 的一期体验版本。

## 体验版本定义

### 体验版本 v0

- 指挥线程能初始化一个 task，并把 `contract + state + context + handoff` 组织完整。
- `dispatch-create` 能把 `Supervisor / Executor / Skeptic` 物化成真实 assignment，而不是口头角色。
- 执行线程能在隔离 worktree 中完成一个切片，并回写 handoff / verification。
- Lead 能通过 `git-integrate-task` 和 closeout/schema 校验把任务收成 `done/archive`。
- `tests/smoke.sh all` 能通过。

### 体验版本还不承诺什么

- 不承诺新的 Hermes runtime、gateway、provider、plugin loader。
- 不承诺自动调度器或“额度恢复自动续跑”已经完整闭环。
- 不承诺默认一定有独立 `Skeptic`；如果当前只能 degraded fallback，必须明示。

## 当前位置

- `dd-hermes-execution-bootstrap` 已完成并归档。
- `dd-hermes-endpoint-router-v1` 已完成追溯回填。
- `dd-hermes-multi-agent-dispatch` 已完成主线收口。
- `dd-hermes-experience-demo-v1` 已完成 execution + integration，证明了一次真实 end-to-end 演示任务可以跑通。

换句话说，DD Hermes 不再只是“具备基础”，而是已经真正跑通了一次从派发到集成的体验版主闭环。

## 当前 blocker

1. 需要建立单一、用户可见的体验入口，而不是继续让体验版真相分散在多份内部文档里。
2. 需要把 phase-done 审计结论固定下来，明确“一期已达到体验版 v0，但尚未 phase done”。
3. 需要把这次体验版证明沉淀成更稳定的演示流程和教材记录。

## 本轮完成后你能看到什么

- 一套明确的多 agent 策略矩阵。
- 一个已经收口的 `dispatch` 主线任务。
- 一次已经跑通并集成回 `main` 的真实体验版演示任务。
- 一份清楚说明“一期体验版本是什么、还不是什么”的中文文档。
