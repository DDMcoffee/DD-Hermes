# 2026-04-16 今日输入整理

这份文档不是当前主线状态页，而是当前仍有效的用户输入归纳。

它回答的是：

- DD Hermes 到底为什么存在
- 这个项目推进时必须先守哪些哲学
- 规划和执行之间哪些要求不能漂移

## 1. 单一结论

- 先哲学，中规划，后执行。
- DD Hermes 的主线是 harness、协议、控制面和工程协作，不是具体业务产品，也不是新的 runtime。
- 对抗式思辨不是装饰，而是协议：复杂问题默认先讨论、留反证位、再进入执行。
- 所有关键判断都必须沉淀成仓库内可审计的 Markdown / JSON 工件，而不是漂在聊天里。

## 2. 四个哲学锚点

### harness 哲学

- 这个 project 专注 Hermes agent / DD Hermes，本质上是 agent harness、协议和框架工程。
- 不聚焦具体应用程序开发。
- 希望把项目做成一个能长期运行、能持续派发 agent 的工程系统，而不是一次性脚手架。
- 需要吸收的基线能力包括：
  - `IMPACT Framework`
  - `Hook 系统`
  - `Skills 工作流`
  - `插件生态`
  - `Lead-Expert 多 Agent 架构`
  - `OpenSpec 规范驱动开发`
  - `安全与权限`
  - `分析与可观测性`

### memory 哲学

- 重点关注 `context / runtime / state / memory engineering`。
- memory 负责长期知识与规律，不能被聊天总结替代。
- 你要求补一个“教材/课本式记录”的 agent，用来记录使用经验、输入链接、结论与规律，并在每天 17:00 产出进展总结。

### Markdown 哲学

- 所有关键判断沉淀成中文 Markdown 文档。
- 给你看的文件优先放在中文命名目录里。
- 文档应该服务指挥线程，而不是零散记在聊天里。
- 对执行线程留下的结果，不能只看聊天，要看：
  - 代码
  - handoff
  - state
  - verification
  - git integration

### Hermes agent 哲学

- 当前线程是唯一指挥线程的候选。
- 对外优先保持一个主线程；额外执行线程只在边界清楚且能降低风险时启用。
- 多 agent 是复杂问题的默认解法，但不要求用户手动管理一堆聊天线程。
- 如果另一个线程已经偏离主线，指挥线程必须收回来裁决。

## 3. 规划要求

- 复杂问题必须优先使用多 agent 讨论，不要单线程直接拍板。
- 当前线程被定义为“指挥线程”。
- 指挥线程负责规划、裁决、收口，而不是随意漂移到执行面。
- 架构问题不要直接编码，要先拆成讨论和执行。
- 你明确要求过：
  - 新建 3 个 agent 做决策讨论
  - 三个 explorer 的结论先收拢成一个清楚形状，再执行
- 每次进入复杂分析或重大判断时，都要明确当前正在走多 agent / 多视角讨论，而不是假装已经有单一答案。

## 4. 执行要求

- 代码执行可以放到独立线程，但是否启用要由指挥线程裁决。
- 不要让我自己跑生成命令；这类命令要由系统直接跑好，再给结果。
- 如果要切到代码线程，不要只给命令，要给现成结果或可直接使用的内容。
- 执行线程跑完后，要检查它的代码情况和收口状态。
- 不要忘记 git 管理；执行切片不能只停在 worktree 或聊天里。

## 5. 验收与终点要求

- 你多次追问过：
  - 项目目标文档和产品文档是不是没有对齐
  - 跑代码线程什么时候才算终点
  - 三层终点是否已经真正制定完
- 执行完成不能只看“代码写完”，还要看：
  - handoff 是否回写
  - state 是否更新
  - verification 是否存在
  - git integration 是否收口
- 你明确要求把工作分支合到主分支，不能只留下悬空执行结果。

## 6. 仍然有效的高优先级输入

- DD Hermes 的主线是框架、协议、控制面，不是具体业务产品。
- 当前线程是唯一指挥线程的候选。
- 是否需要额外执行线程，必须由“边界是否清楚、是否会跑偏”来决定。
- 多 agent 是复杂问题的默认解法，不是装饰。
- 模型和执行线程选择不能凭感觉改口，要能被讨论、被裁决、被解释。
- 所有关键要求都应该持续沉淀到 `指挥文档/` 中，而不是只存在聊天上下文里。
