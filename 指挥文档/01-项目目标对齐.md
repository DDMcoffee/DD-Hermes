# DD Hermes 一期项目目标对齐

## 北极星目标

DD Hermes 一期不是做一个新的 agent runtime，也不是做一个新的 CLI。

它的目标只有一个：

- 在 Codex IDE 里搭出一个可长期运行的多 agent 工程 harness，让指挥线程、执行线程、memory、state、context、git worktree 和 verification 都有明确协议。
  - 现在进一步收敛成：在单线程默认模型里，用恒定的 `Product Anchor` 和 `Quality Anchor` 持续锚定目标与质量。

## 本期产物

- 单线程默认 + 角色切换协议，以及两个恒定锚点的治理协议。
- `contract + state + context + handoff + worktree` 的显式协作面。
- hooks、模板、OpenSpec、memory 治理、git 管理和 smoke test。
- 可以按需派发 execution slice，而不是依赖聊天历史维持工程状态。

## 明确不做

- 新的 Hermes runtime、gateway、provider、plugin loader。
- 向量数据库、复杂 RAG、独立产品化 UI。
- 把“聊天总结”误当作 memory 系统。

## 文档分层

当前项目里有三类文档，不能再混写：

1. 项目目标文档
   - 回答“DD Hermes 一期为什么存在，finish line 是什么”。
   - 主文档：本文件。
2. 协议文档
   - 回答“线程、worktree、state、memory、verification 怎么协作”。
   - 主文档：`README.md`、`AGENTS.md`、`docs/`。
3. 任务实例文档
   - 回答“某个 task 当前做到哪、谁负责、下一步是什么”。
   - 主文档：`workspace/`、`openspec/`。

## 当前失配点

- `README.md` 说的是项目级目标。
- `workspace/contracts/dd-hermes-execution-bootstrap.md` 和 `openspec/proposals/dd-hermes-execution-bootstrap.md` 说的是 bootstrap 任务级目标。
- 这两层之前没有一份中文文档把“项目目标”和“任务目标”拆开，所以看起来像产品文档和工程协议文档没对齐。

## 对齐后的单一判断

- 项目级 finish line：DD Hermes 一期 harness 本身成型。
- 任务级 finish line：某个 task 的 contract / handoff / code / verification / git integration 收口完成。
- 线程级 finish line：当前执行 slice 结束并回写。

以后如果再讨论“目标不清楚”，默认先回到这份文档，再往下看任务实例。
