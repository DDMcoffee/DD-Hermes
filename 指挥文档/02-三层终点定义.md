# DD Hermes 执行与完成模型

## 总原则

DD Hermes 的“完成”不能只看某个代码线程有没有停下来，必须分三层判断：

1. `execution slice done`
2. `task done`
3. `phase done`

如果只满足前一层，就不能冒充后一层。

## 第一层：Execution Slice Done

### 责任人

- `Executor`

### 进入条件

- 已有 `task_id`
- 已有 `contract + state + context + handoff + worktree`
- 若 `discussion.policy == 3-explorer-then-execute`，必须已有 `synthesis_path`
- `lease.status` 不是 `paused`

### 退出条件

- 改动只落在指定 worktree 或当前切片边界内
- 写回 handoff
- 写回 verification 证据
- 形成 execution commit

### Done Definition

- 改动已经提交成一个可评审切片
- `workspace/handoffs/*expert-to-lead.md` 已写清决策、风险、验证、下一步
- `workspace/state/<task_id>/state.json` 已记录验证结果和 commit 锚点

### 阻塞条件

- 事实不清
- 越权改动
- 需要指挥线程裁决的方向性问题
- 验证连续失败且没有新的本地缩小路径

## 第二层：Task Done

### 责任人

- `Lead`

### 进入条件

- 至少一个 execution slice 已完成
- task contract、proposal、handoff、state 都存在

### 退出条件

- execution slice 被集成回主控制面
- 重新跑任务级验证
- task 的 acceptance 都满足

### Done Definition

- mainline 或指挥分支拥有完整实现
- integration commit 已形成
- `state.status` 已切到与真实任务一致的完成态
- OpenSpec 至少推进到和任务真实状态一致的阶段

### 阻塞条件

- execution branch 还未收口
- handoff 和真实代码不一致
- integration 后测试回退

## 第三层：Phase Done

### 责任人

- `Lead / Judge`

### 进入条件

- 若干个 task 已完成
- 项目级目标文档、协议文档、任务实例文档相互一致

### 退出条件

- 当前阶段范围内的 harness 能稳定支持下一轮任务派发
- 文档、脚本、模板、验证、git workflow 不再互相打架

### Done Definition

- 项目目标文档、协议文档、任务实例文档已经对齐
- 多 agent 流程可以稳定重复
- `main` 上保留的是可继续派发的基线，而不是实验残片

### 阻塞条件

- 项目目标还依赖口头解释
- 关键脚本只在某个 worktree 内有效
- 文档和代码对同一流程给出不同答案

## 实现阶段怎么进入

- 默认是当前线程内角色切换，不默认新开第二聊天线程。
- 只有当写集清晰、切片边界稳定、并且独立 worktree 能真实降低风险时，才开 execution slice。
- `scripts/execution-thread-prompt.sh` 现在只是兼容入口和检查清单，不再等于“必须新开一个聊天线程”。

## 实现阶段何时停止

- 达到 `execution slice done` 就停，不继续冒充 `task done`
- 如果发现目标漂移、锚点冲突、权限边界变化或架构歧义，立即退回当前主线程裁决
- 若当前工作其实已经变成文档裁决、状态回填或归档，不再假装它还是 execution slice

## 验收顺序

1. 先验 `execution slice done`
2. 再验 `task done`
3. 最后验 `phase done`

任何跳层宣称“完成”，都算协议错误。
