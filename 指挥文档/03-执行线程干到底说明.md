# 执行线程干到底说明

## 目标

- 不再继续补新的零散脚手架，而是做一次真正的端到端体验演示任务。
- 证明 DD Hermes 已经能把一个任务从派发、执行、验证、集成一路收成 `done/archive`。

## 终点

- 新建一个边界清晰的小型真实任务，并为它生成完整的 `contract + state + context + handoff + worktree`。
- 执行线程在隔离 worktree 中完成实现，形成 execution commit，并回写 handoff / verification。
- Lead 能通过现有 `git-integrate-task`、schema 校验和 archive 流程把它收成 `done/archive`。
- 形成 execution commit，并把验证证据回写给指挥线程。

## 禁止事项

- 不要修改项目级 policy。
- 不要直接在主工作区提交 execution slice。
- 不要把未验证的猜测写进 handoff 或 state。
- 一旦发现 finish line 已经从“代码实现”变成“项目目标裁决”，立即停回指挥线程。

## 执行顺序

1. 先把三层终点文档拆成可执行 schema 和 closeout 模板。
2. 用现有 harness 建立真实任务工件，而不是扩写新的控制面能力。
3. 在 worktree 内完成实现、验证、handoff 和 execution commit。
4. 把结果交回指挥线程做 integration / archive。

## 最小实现范围

- 一个真实的小型实现切片
- 现有 harness 所要求的全部任务工件
- 至少一条技术验证链路和一条任务级 schema 校验链路

## 交付物

- 真实任务代码改动
- execution commit
- verification 证据
- expert handoff

## 回写要求

- 回写 `workspace/handoffs/*expert-to-lead.md`
- 回写当前 task state 的 verification / git 锚点
- 回写必要的 note，说明是否已经达到 `execution slice done`

## 何时停

- 当真实任务已经达到 `execution slice done`，且 execution commit、handoff、verification、state 锚点都完成时停止。
- 遇到新的架构歧义、项目级目标冲突、越权修改需求或硬性验证阻塞时停止，并回到指挥线程。
