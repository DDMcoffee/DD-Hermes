# 执行线程干到底说明

## 目标

- 把 `dd-hermes-execution-bootstrap` 从“验证过的 worktree 改动”推进到真正可集成的 execution slice。
- 保证模板驱动的 sprint bootstrap、shared repo root 访问和 smoke coverage 都在主线协议下成立。

## 终点

- execution worktree 中的改动提交成 execution commit。
- lead 能在主控制面无歧义地集成这批改动。
- 所有验证命令在集成后仍然通过。

## 禁止事项

- 不要修改项目级 policy。
- 不要直接在主工作区提交 execution slice。
- 不要把未验证的猜测写进 handoff 或 state。
- 一旦发现 finish line 已经从“代码集成”变成“项目目标裁决”，立即停回指挥线程。

## 执行顺序

1. 在分配的 worktree 内收口本轮代码改动。
2. 运行最小充分验证，确认 `sprint-init / context/state / git / smoke` 相关链路通过。
3. 写回 expert handoff，明确本轮文件、验证、风险和下一步。
4. 形成 execution commit。
5. 把 commit 锚点和验证证据回写给 lead。

## 交付物

- execution commit
- expert handoff
- verification 证据
- 如有必要，补充 exploration 或 task 级说明

## 回写要求

- 回写 `workspace/handoffs/*expert-to-lead.md`
- 回写 `workspace/state/dd-hermes-execution-bootstrap/state.json` 的 verification / git 锚点
- 回写必要的 note，说明本轮是否已经达到 `execution slice done`

## 何时停

- execution commit 已形成，且 handoff、verification、state 锚点都完成时停止。
- 遇到新的架构歧义、项目级目标冲突、越权修改需求或硬性验证阻塞时停止，并回到指挥线程。
