# DD Hermes 产品介绍与使用说明

这页不是历史总结，也不是当前任务状态页。它只回答三件事：

1. DD Hermes 是什么产品
2. 它对用户暴露的稳定表面是什么
3. 第一次接手或继续推进时，最短应该怎么操作

## 1. 产品定位

### 一句话

DD Hermes 是一个面向 Codex IDE 的工程 harness / 控制面，用显式工件把复杂 agent 工作从“靠聊天协调”变成“可规划、可执行、可验收、可归档”的工程流程。

### 核心表面

DD Hermes 对外暴露的不是一个新的 runtime，而是一套稳定的工程协作表面：

- 运行态入口：`./scripts/demo-entry.sh`
- 任务控制面：`contract + state + context + handoff`
- 执行隔离：`worktree`
- 完成闭环：`verification + closeout + quality-gate`

### 它不做什么

明确不做：

- 新的 Hermes runtime
- 新的 provider / gateway / plugin loader
- 用聊天记录替代仓库内控制面
- 把用户变成多个线程的手工调度员

## 2. 适合什么场景

DD Hermes 最适合这几类任务：

1. 需要先压实目标，再拆分执行
2. 需要留下明确验收和归档证据
3. 真实业务改动在 target repo，而控制面要留在当前仓
4. 对外只想保留一个主线程，但对内仍要有多角色和隔离 worktree

如果只是一次性微改动，DD Hermes 也能用，但价值主要体现在复杂工程任务上。

## 3. 产品为什么这样设计

### 对外单入口，对内多角色

用户默认只看一个主线程。

内部可以切换 `Lead / Explorer / Executor / Skeptic / Product Anchor / Quality Anchor`，但这些复杂度应该留在协议层，不应该要求用户手工维护第二聊天线程。

### 真相落在工件，不落在回忆

DD Hermes 尽量让下面这些判断都能从仓库里直接读出来：

- 任务边界
- 当前控制面状态
- 最新上下文
- 执行交接
- 验证证据
- 是否真正完成

如果这些问题必须靠聊天历史回答，说明控制面还没有收口好。

### 跨仓是默认运行形态

大多数真实业务改动不在 DD Hermes 仓本身，而在 target repo。

因此默认分工是：

- DD Hermes 仓
  - 维护 `contract / state / context / handoff / closeout / memory`
- target repo
  - 落业务代码、跑测试、做集成

## 4. 对用户暴露的稳定操作路径

### 情况 A：只想知道现在是什么状态

先运行：

```bash
./scripts/demo-entry.sh
```

再读：

1. `指挥文档/06-一期PhaseDone审计.md`
2. `指挥文档/04-任务重校准与线程策略.md`

这一步只回答：

- 现在能不能用
- 有没有 active mainline
- 最近 proof 是什么
- 下一步是继续、等待，还是立新主线

### 情况 B：继续推进当前主线

如果入口显示存在 active mainline，最短路径是：

1. 读 `workspace/contracts/<task_id>.md`
2. 读 `workspace/state/<task_id>/state.json`
3. 读 `workspace/handoffs/<task_id>-lead-to-<expert>.md`
4. 如果已有执行结果，再读对应 `workspace/closeouts/`

最常用命令：

```bash
./scripts/state-read.sh --task-id <task_id>
./scripts/context-build.sh --task-id <task_id> --agent-role commander
./hooks/quality-gate.sh --event Stop --state workspace/state/<task_id>/state.json
```

### 情况 C：当前没有 active mainline

不要直接凭聊天记忆开新任务。

先回到：

1. `指挥文档/04-任务重校准与线程策略.md`
2. 最近 proof 的 `openspec/archive/*.md`

只有在 repo evidence 明确支持下一条 bounded task package 时，才重新进入：

- 新 contract
- 新 state
- 新 handoff
- 新 context
- target repo worktree

### 情况 D：本次执行已经做完，准备收口

真正完成至少要同时看到：

- fresh verification
- closeout
- state writeback
- `quality-gate`
- integration / archive 决策

“代码写完”只是执行结果，不等于任务完成。

## 5. 一次典型工作怎么走

### 第一步：判断当前状态

```bash
./scripts/demo-entry.sh
```

### 第二步：决定是继续还是新开

- 有 active mainline：进入现有任务包
- 没有 active mainline：回到 `04` 和最近 archive，再决定是否启动新主线

### 第三步：装配控制面

最小集合：

- `contract`
- `state`
- `context`
- `handoff`

### 第四步：在 target repo 执行

真实业务改动默认在 target repo 的隔离 worktree 中进行，避免主工作区中间态污染。

### 第五步：回写证据并收口

- verification
- closeout
- quality review
- archive / integration

## 6. 最常用命令

```bash
./scripts/demo-entry.sh
./scripts/state-read.sh --task-id <task_id>
./scripts/context-build.sh --task-id <task_id> --agent-role commander
./scripts/dispatch-create.sh --task-id <task_id>
./scripts/check-artifact-schemas.sh --task-id <task_id>
./hooks/quality-gate.sh --event Stop --state workspace/state/<task_id>/state.json
bash tests/smoke.sh schema
```

## 7. 使用时的几个硬判断

### 一个主线程，对外尽量简单

用户默认只看当前线程。

内部可以有多角色、多 worktree、多证据链，但不要把复杂度泄漏成用户必须手工管理的第二聊天线程。

### 控制面诚实高于“看起来一直在推进”

如果当前没有 active mainline，就明确写没有；如果任务阻塞，就明确写阻塞；如果只是 local residue，就不要把它包装成 committed successor。

### README 和指挥文档不承载运行态真相

当前主线、最新 proof、下一步决策，都优先从运行态入口和任务级工件读取，不直接写死在 landing 文档里。

### 完成不是“写过代码”

真正完成至少要能从仓库工件里直接看到：边界、执行、验证、closeout 和 gate verdict。
