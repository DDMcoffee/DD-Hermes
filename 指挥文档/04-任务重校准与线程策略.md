# DD Hermes 当前任务重校准与线程策略

## 当前事实

- `main` 上已经有 endpoint/schema、coordination router、dispatch/create、thread-switch gate、lease-check 等控制面能力。
- `dd-hermes-execution-bootstrap` 已经归档。
- `dd-hermes-endpoint-router-v1` 已完成追溯回填并归档。
- `dd-hermes-multi-agent-dispatch` 已完成追溯回填并归档。
- 当前主线不再缺 router/dispatch 能力，缺的是一次面向体验版本的真实演示任务。

## 重校准后的单一目标

当前阶段不再把主目标写成“继续补 endpoint/schema/router”。

当前唯一主目标是：

- 用已经收口的控制面能力跑出一次端到端体验版演示任务，并继续维持不跑偏的线程纪律。

换句话说，当前重点已经从“收口和治理”切到“用现有能力做体验版证明”，不是继续平行扩写功能面。

## 不该继续混入主线的内容

- 新的 Hermes runtime、gateway、provider、plugin loader。
- 与当前控制面收口无关的功能扩展。
- 把不同 task 的追溯回填混成一个大 task。
- 在没有明确切片边界时同时开多个执行线程。

## 任务拆分

### Task 0：定义体验版演示切片

#### 目标

- 定义一个边界清晰、足够小、能完整走通 harness 的真实任务切片。

#### 边界

- 这是指挥线程的规划任务，不是大范围功能扩展。
- 不新增新的控制面能力，只消费现有能力完成一次真实任务。

#### 完成定义

- 演示任务必须有唯一 `task_id`、明确 acceptance 和清楚的非目标。
- 演示任务不能再把“补脚手架”伪装成“体验版”。

### Task 1：执行一次单 execution thread 演示

#### 目标

- 派一个有边界的 execution thread，让它在隔离 worktree 中完成真实任务、回写 handoff、形成 execution commit。

#### 边界

- 只开一个 execution thread，不开并跑第二线程。
- 目标是体验版演示，不是扩写新的调度器或运行时。

#### 完成定义

- 真实任务达到 `execution slice done`。
- 指挥线程能据此完成 integration / archive。

### Task 2：做 phase-done 审计

#### 目标

- 在体验版演示完成后，再判断 DD Hermes 一期距离 `phase done` 还差什么。

#### 边界

- 只审计项目级 finish line，不新增功能。
- 产出 blocker 列表和下一轮唯一主目标。

#### 完成定义

- 能明确回答“一期还差哪些 blocker”。
- 如果 blocker 只剩 task-bound traceability，则继续 command thread 收口。
- 如果 blocker 是真实代码缺口，则创建下一条单独 task，再派一个 execution thread。

## 线程策略

## 多 Agent 使用准则

- 治理、归档、trace 回填、finish line 校准这类任务，由指挥线程单线程完成，不额外派 execution thread。
- 单一实现切片，采用 `Lead + Executor`；前提是 `contract + state + context + handoff + worktree` 已齐备。
- 架构、控制面、权限/安全边界、状态机、线程模型这类任务，默认升级为 `Lead + Executor + Skeptic`。
- 需要两条独立证据链的任务，默认升级为 `Lead + Executor A + Executor B + Skeptic`，而不是单执行者自证。
- 若 `Skeptic` 不独立，必须把任务标成 `degraded`，并把“体验版本未达到独立监督”列为已知缺口。

## 结论

- 现在不需要两个活跃线程并行推进主线。
- 当前更合理的形态是：`一个指挥线程 + 零个或一个按需执行线程`。

## 什么时候不该开第二线程

- 当前主目标还是治理、收口、裁决。
- 另一个线程留下的是回填工件，不是正在推进的核心实现。
- 当前 task 没有两个互不冲突、边界清晰、都需要立刻编码的执行切片。

## 什么时候才开 execution thread

- 已经有明确 task id。
- 已经有 contract、state、handoff、context。
- `discussion.synthesis_path` 存在。
- `lease` 未暂停。
- 目标是一个真实的代码切片，而不是文档裁决或任务清理。

## 什么时候才允许两个 execution threads

- 只有同时满足下面条件才允许：
- 两个切片写集不重叠。
- 两个切片都有稳定接口和独立 acceptance。
- `scale_out_recommended=true`。
- 指挥线程还能维持单一裁决面。

否则，第二线程带来的价值小于漂移成本。

## 当前裁决

1. 保持这个线程作为唯一指挥线程。
2. 不开第二个并跑主线线程。
3. 先由指挥线程定义体验版演示切片。
4. 然后只派一个 execution thread 完成真实任务。
5. 演示任务结束后，再做 phase-done 审计。

## 对另一个线程的态度

- 它不是不能用。
- 但在当前阶段，不应把另一个线程当成长期并跑主线，而应该把它当成按需调用的执行面。

所以，正确处理方式不是让多个线程长期自转，而是由指挥线程定义单一体验任务，再按需打开和关闭 execution thread。
