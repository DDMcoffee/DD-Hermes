# DD Hermes 当前任务重校准与线程策略

## 当前单一判断

- 一期已经 `phase done`
- `dd-hermes-independent-quality-seat-v1` 已归档
- 当前 phase-2 唯一主线是 `dd-hermes-quality-seat-escalation-rules-v1`
- 当前线程继续作为唯一默认指挥线程
- phase-2 当前主线现在已经有正式 `contract + state + proposal + design + task + handoff + exploration + decision synthesis`

所以，这一阶段的主目标不是“再补一个外部执行线程”，而是把 phase-2 的“独立质量位”压成一个可执行、可验证的真实主线。

## 当前主线到底做什么

`dd-hermes-independent-quality-seat-v1` 的目标只有一条：

- 让维护者在 `state / dispatch / context / gate` 任一入口都能看清：当前质量位到底是 `independent` 还是 `degraded`。

不在本主线里的内容：

- 新的 Hermes runtime、gateway、provider、plugin loader
- 为了“像多 agent”而额外制造新的聊天线程
- 把治理任务和无关功能扩展混成一个大 task

状态：

## 现在需不需要第二线程

不是默认需要。

当前更合理的形态仍然是：

- `一个主线程 + 线程内多角色切换 + 按需隔离 worktree`

只有同时满足下面条件，才值得开一个 execution slice：

- 已有明确 `task_id`
- 已有 `contract + state + context + handoff + worktree`
- 已有 `synthesis_path`
- 写集边界稳定
- 额外 worktree 能真实降低冲突或回归风险

如果当前工作还是 planning、裁决、归档、审计、文档收口或 trace 回填，就不该开第二线程。

只有当下一轮真的需要隔离写集时，才考虑把后续 execution slice 外派到独立 worktree。

## 多 Agent 怎么用

- 架构、控制面、线程模型、权限/安全边界，默认先走 `3-explorer-then-execute`
- 这里的“多 agent”首先是当前线程内的协议角色与内部 explorer 收敛，不要求用户手动管理多个聊天线程
- 用户只保留当前线程作为对话面；其他线程如果存在，只作为系统内部执行面或历史来源，由系统自行生成、吸收、关闭

## 管理职责现在落在哪里

不存在一个必须长期在线的“管理 agent 的 agent”。

当前真实设计是：

- `Supervisor / Product Anchor` 负责目标压缩、主线裁决和状态推进
- `Skeptic / Quality Anchor` 负责独立审查或显式降级
- 管理真相落在 `state + context + gate`，不是落在一个永久聊天线程上

## 当前裁决

- `dd-hermes-independent-quality-seat-v1` 现在是已归档的 proof task，不再承载新实现
- 当前 phase-2 主线已经切到 `dd-hermes-quality-seat-escalation-rules-v1`
- 当前下一步不是继续扩旧 task，而是把升级规则主线压成真正可执行的第一条实现边界
- 当前仍不把第二线程当默认主线；后续只有在写集独立且回归风险更低时才外派
