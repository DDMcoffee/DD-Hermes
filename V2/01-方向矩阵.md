# DD Hermes V2 方向矩阵

7 个候选支柱 + 7 Explorer 视角头脑风暴的 Synthesis 结果；每条附 `provable_need_sketch` 与 `target_repo_candidates`，供将来立 slice 时直接复用。本文件不是"即将立案的 PRD"，而是**候选池**。

## 0. 如何读这张表

- `codex_app_leverage`：这个支柱如何直接帮助"更好使用 Codex APP"（落到具体交互场景）
- `knowledge_deposit_path`：在"用户 know-how → skill → stack"蒸馏链上的位置
- `provable_need_sketch`：当前仓库里已观察到的真实瓶颈指针，供未来 `Self-Reference Ops` 的 `provable_need` 硬规则直接引用
- `target_repo_candidates`：候选真实 target repo 池，**不现在绑定**，立 slice 时按 `AGENTS.md → Cross-Repo Execution` 选型
- `self_ref_risk`：是否触发 `Self-Reference Ops` 硬规则（`low` / `medium` / `high=deferred`）
- `votes`：7 Explorer 视角的综合评分（详见附录）

## 1. 候选支柱

### 1.1 `v2-skill-stack` ★★★★★（均分 4.83，首批）

- **intent**：把 `memory/self/*` 与归档 task 里沉淀的 know-how 蒸馏成 `memory/skills/*.md`，带 progressive disclosure（只暴露 `name + description + when-to-use`，用时再加载完整内容）
- **codex_app_leverage**：用户在 Codex 里开新任务时，Agent 可以零 context 开销地调用过往教训（例如"遇到 gitignore-protected 路径就改用 `/tmp + cp`"）
- **knowledge_deposit_path**：蒸馏链 `memory/self → memory/skills → Stack` 的 **2→3 位置**，是整个 V2 的蒸馏枢纽
- **provable_need_sketch**：
  - `memory/self/recalibration-2026-04-18-learnings.md` 记录 5 条重复教训（gitignore 写文件、parallel write_to_file 部分失败、stdin mode closeout fallback 等），但没有可被下一轮主动调用的 skill 条目，只能靠人脑 recall
  - `openspec/archive/` 40+ 条归档，`memory/self/` 只 2 条 → 蒸馏率 < 5%
- **v1_dependency**：`memory/self/*`、`memory-write.sh`、`memory-read.sh`
- **github_anchor**：
  - Anthropic Claude Skills 官方规范（progressive disclosure）
  - `alirezarezvani/claude-skills` 232+ skills
  - `muratcankoylan/Agent-Skills-for-Context-Engineering`（Peking Univ 2026，6 模块隔离 + append-only JSONL）
- **docx_anchor**：3.4 积累之道 / 终章"Skill 之后的蒸馏"
- **target_repo_candidates**：`self`（DD Hermes 自身，蒸馏自己的 know-how 是第一步）
- **self_ref_risk**：**medium** —— 是 harness 自指但 `provable_need` 明确（memory/self 稀疏是观测事实）
- **drift_risk**：把 skill 做成"新一套文档协议"而非 know-how 蒸馏；progressive disclosure 被忽略导致每个 skill 都被完整加载
- **first_slice_shape**：S2 slice；产出 `memory/skills/SKILL-SCHEMA.md`（格式规范）+ 3 条种子 skill（从 recalibration learnings 提取）+ `scripts/skill-list.sh` 只读列表 + 1 个验证点（`bash tests/smoke.sh schema` 覆盖 skill schema）

- **v1_dependency**：`context-build.sh`、`successor-evidence-audit.sh`、`demo-entry.sh`

## 4. 争议项（Explorer 之间无法完全对齐的点）

### 4.1 `v2-multi-codex-fleet` 的优先级

- **Explorer δ（Codex UX）** 给 ★4：认为用户多项目并行是真刚需
- **Explorer γ（V1 痛点）** 给 ★1：仓库里无证据支持多项目需求
- **Explorer η（成本）** 给 ★1：可能需要 runtime 层，成本高
- **Synthesis 结论**：保留 `deferred`，但记录**升级触发条件** = "用户 Codex APP 实际出现 ≥ 2 个并行长期项目 + 跨项目 skill 复用真实需求"；不等同于 `wont-fix`

### 4.2 Skill 到底归 Goal 还是 Context？

- **Explorer β（GitHub）** 按 Anthropic 官方归 `Goal`
- **Explorer γ（V1 痛点）** 按 DD Hermes 落地语义归 `Context`
- **Synthesis 结论**：V2 在哲学总纲 §2.3 明确归 **Context 的可执行投影**；不否定 docx / Anthropic 的抽象，但落地按 DD Hermes 工件语义

### 4.3 "大力出奇迹"是否默认吸收？

- **Explorer α（docx）** 倾向吸收（docx 8.3 明确主张）
- **Explorer η（成本）** 反对（违反 `Dangerous Ops` 资源边界精神）
- **Explorer ζ（离经叛道）** 认为可以在 Co-spark 语境有限开放
- **Synthesis 结论**：**不作为 V2 默认**；仅在 `v2-co-spark` opt-in + contract 显式声明 `cost_budget` 的语境下有限允许（哲学总纲 §8.1）

### 4.4 V2 是否需要新的 daemon runtime？

- **Explorer ζ（离经叛道）** 提出 Compound Loop 自动化可能需要事件驱动 daemon
- **Explorer γ（V1 痛点）** 反对：V1 明确"不做 runtime"是稳定边界
- **Explorer η（成本）** 反对：daemon 需要长期在线成本
- **Synthesis 结论**：V2 **先做脚本化**（`scripts/compound-review.sh` 手动或 cron），不做 daemon；留给 V3 远景

## 附录 B：各视角主要发现

- **α（docx 哲学）**：五步闭环缺 Scout + Co-spark，是 V2 最明确的空白
- **β（GitHub 明星）**：Skill Stack + OpenSpec 是 2026 最主流方向；Peking Univ `Agent-Skills-for-Context-Engineering` 提供了完整的 6 模块 + JSONL 参考
- **γ（V1 真实瓶颈）**：`memory/self` 稀疏（2 条 vs 40+ 归档）是最硬的 provable_need
- **δ（Codex UX）**：用户每天在 Codex 里的三个摩擦点 = 立任务前补 context / 读 contract 看是否 pass / 看昨天沉淀了什么 → 分别对应 Scout、Decision Gate、Textbook
- **ε（Compound）**：蒸馏链条 `memory/task → memory/self → memory/skills → Stack` 的瓶颈在 2→3 位置（主动蒸馏缺失）
- **ζ（离经叛道）**：只有 `v2-multi-codex-fleet` 真正挑战 V1，但无 provable_need；其他支柱都在 V1 框架内自然扩展
- **η（成本 ROI）**：Co-spark 的 N 倍 token 成本 + Fleet 的长期 daemon 成本是两个红线

## 附录 C：本文件的性质

- 本文件是**候选池**，不是 PRD；不是立案承诺
- 任一支柱在立 slice 时必须**独立**附 `provable_need`，引用这里的 `provable_need_sketch` 作为起点但需要更新到立案当时的真实仓库状态
- `self_ref_risk=high` 的条目默认 `deferred`；`AGENTS.md → Self-Reference Ops` 明确"`deferred` 条目每季度回顾一次，仍无 `provable_need` → 归档为 `wont-fix`"
- 本文件允许 V2 后续会话追加新候选支柱，但修改既有条目的优先级或 `self_ref_risk` 必须有仓库 evidence 支持
