# DD Hermes V2 方向矩阵

7 个候选支柱 + 7 Explorer 视角头脑风暴的 Synthesis 结果；每条附 `provable_need_sketch` 与 `target_repo_candidates`，供将来讨论时作为起点。**本文件不是"即将立案的 PRD"，而是候选池 + 讨论记录**。所有字段都是**讨论性假设**，不是立案承诺；任一支柱立 slice 时必须按 `AGENTS.md → Self-Reference Ops` 重新附上当时的真实 `provable_need`。

## 0. 如何读这张表

- `分层`：`直接用户收益` / `harness enabler` / `边缘` 三类；直接用户收益优先，harness enabler 附更高证据门槛（见 §1 前言）
- `证据强度`：`A 直接证据` / `B 间接证据` / `C 待补证`（见 §1 前言）
- `codex_app_leverage`：这个支柱如何直接帮助"更好使用 Codex APP"（落到具体交互场景）
- `knowledge_deposit_path`：在"用户 know-how → skill → 更稳定的 know-how 库"蒸馏链上的位置（注：本文件原用的"Stack"一词已收敛为"更稳定的 know-how 库"，避免与 GCC 并列成模糊新层级，见 §7）
- `provable_need_sketch`：当前仓库里已观察到的真实瓶颈指针，供未来 `Self-Reference Ops` 的 `provable_need` 硬规则直接引用
- `target_repo_candidates`：候选真实 target repo 池，**不现在绑定**，立 slice 时按 `AGENTS.md → Cross-Repo Execution` 选型
- `self_ref_risk`：是否触发 `Self-Reference Ops` 硬规则（`low` / `medium` / `high=deferred`）
- `if-slice-opened shape`（原 `first_slice_shape`）：**假设性草图**，不是立案承诺
- `votes`：7 Explorer 视角的 Synthesis 评分，仅作讨论顺序参考，非测量（免责见附录 A）

## 1. 候选支柱

### §1 前言：分层与证据强度

**分层（按第三方审核 3.3 建议）**：V2 北极星是"让用户更好使用 Codex APP + 沉淀用户经验"，直接用户收益方向应前置，harness enabler 附更高证据门槛。7 条候选里**只有 `v2-textbook-daily` 明确是直接用户收益**，其余 6 条都偏 harness enabler —— 这是第一轮头脑风暴被 docx 5 步闭环与 GitHub 明星项目带偏的痕迹；修订建议：立 slice 时优先验证直接用户收益方向，harness enabler 类支柱在有直接用户痛点证据前不升级优先级。

**证据强度**：
- **A**：有当前仓库里的直接 evidence 指向该支柱要解决的瓶颈（例：`memory/self` 稀疏 2/19、`docs/textbook-agent.md` 声明未落地）
- **B**：有间接证据但方向仍然合理（例：`successor-evidence-audit` 暴露立案前 context 需求）
- **C**：当前无直接证据或 evidence 方向偏，属于**待补证 / 实验项**，立 slice 前必须补齐直接证据

7 条候选在下方**保留第一轮头脑风暴的原评分顺序**（未按分层重排，避免引入二次漂移），但每条顶部标注 `分层` + `证据强度`，读者按这两个标签理解优先级即可；§3 会按二层 + 证据强度给出**讨论顺序**（不是排期）。

## 3. 讨论顺序（不是排期）

**本节明确不是 roadmap** —— 以下仅是"若开新讨论该从哪一条入手"的建议，不承诺任何立案、不锁定任何执行顺序。真正立 slice 必须按 `AGENTS.md → Self-Reference Ops` 单独举证。

**按分层 + 证据强度排**：直接用户收益先于 harness enabler；A 证据先于 B 先于 C。

## 4. 争议项（Explorer 之间无法完全对齐的点）

### 4.1 `v2-multi-codex-fleet` 的优先级

- **Explorer δ（Codex UX）** 给 ★4：认为用户多项目并行是真刚需
- **Explorer γ（V1 痛点）** 给 ★1：仓库里无证据支持多项目需求
- **Explorer η（成本）** 给 ★1：可能需要 runtime 层，成本高
- **Synthesis 结论**：保留 `deferred`，但记录**升级触发条件** = "用户 Codex APP 实际出现 ≥ 2 个并行长期项目 + 跨项目 skill 复用真实需求"；不等同于 `wont-fix`

### 4.2 Skill 到底归 Goal 还是 Context？

- **Explorer β（GitHub）** 按 Anthropic 官方归 `Goal`
- **Explorer γ（V1 痛点）** 按 DD Hermes 落地语义归 `Context`
- **Synthesis 结论**：V2 在哲学总纲 §2.3 明确归 **Context 的可执行投影**；不否定 docx / Anthropic 的抽象，但落地按 DD Hermes 工件语义

### 4.3 "大力出奇迹"是否默认吸收？

- **Explorer α（docx）** 倾向吸收（docx 8.3 明确主张）
- **Explorer η（成本）** 反对（违反 `Dangerous Ops` 资源边界精神）
- **Explorer ζ（离经叛道）** 认为可以在 Co-spark 语境有限开放
- **Synthesis 结论**：**不作为 V2 默认**；仅在 `v2-co-spark` opt-in + contract 显式声明 `cost_budget` 的语境下有限允许（哲学总纲 §8.1）

### 4.4 V2 是否需要新的 daemon runtime？

- **Explorer ζ（离经叛道）** 提出 Compound Loop 自动化可能需要事件驱动 daemon
- **Explorer γ（V1 痛点）** 反对：V1 明确"不做 runtime"是稳定边界
- **Explorer η（成本）** 反对：daemon 需要长期在线成本
