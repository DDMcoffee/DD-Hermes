# DD Hermes V2 哲学总纲

## 0. 本文件的四层来源声明（第三方审核 3.4 要求）

本文件所有主张按下列四层来源区分；任一主张都可被追问"你是哪一层"：

- 🎯 **用户明确输入**：用户对 V2 的直接要求或取向。来源例：`V2 是下一代大版本扩展空间 / 北极星是更好使用 Codex APP + 沉淀个人知识 / 不要一口气把哲学 V2 docx 全盘吸收`
- 📂 **仓库现有真相**：本仓库里可观测的事实，通过 `./scripts/demo-entry.sh` / `openspec/archive/` / `memory/self/` / `docs/` 等直接验证。来源例：`openspec/archive/ 19 条 / memory/self/ 2 条 / docs/textbook/daily/ 3 条`
- 📘 **外部参考筛选后吸收**：`docx《Agentic 之道 V2》`、GitHub 明星项目、Anthropic Skills 等外部材料，**筛选后**的引用。筛选与拒绝动作见 §8
- 🔀 **头脑风暴 / 多 Explorer 结果**：Synthesis 过程的内部讨论产物，**强度弱于上述三层**；未经直接证据补齐前不升级为协议。方向矩阵里的 7 条候选、评分、分层标签都属于这一层

**关键原则**：🎯 和 📂 优先于 📘，📘 优先于 🔀。V2 不是 docx 的本地翻版，也不是头脑风暴的直接执行表。

## 1. V2 是什么

### 1.1 主骨架 + 映射语言（第三方审核 4.1 要求澄清）

V2 的**主骨架仍是 V1 的工件八件套 + memory/**：`contract / state / context / handoff / worktree / closeout / exploration / archive` + `memory/{task,self,user,views}`。所有立 slice、验收、归档都走这套；V2 不新造一套主骨架。

V2 在这之上只叠加：

- **GCC 映射语言**（见 §2）：`Goal / Context / Constraints` 三元组作为对外沟通和分析的**辅助词汇**，**不是协议骨架**。任何文档谈到 GCC 时都可以被翻译回 V1 工件（翻译表见 §2.2）
- **面向 Codex APP 使用场景的候选方向**（见 `01-方向矩阵.md`）：7 条候选支柱按 `分层 / 证据强度` 组织；不是 roadmap

### 1.2 一句话定位

DD Hermes V2 = **面向 Codex APP 的工程马具** + **用户个人知识蒸馏系统**。

工程马具负责把 Codex 的生成能力约束到可验收、可归档、可复用的工程结果上；知识蒸馏系统负责把每一次使用过程的 know-how 沉淀成可再调用的 Context，形成复利。

两者不是并列，是**同一件事的两个面**：套马具的过程必然产生 know-how，沉淀 know-how 又反过来让马具更懂用户。这呼应 docx 终章 "你积累的 Context 不会贬值" 的结论。(📘 来源：docx 终章；已确认与 V1 `memory/{task,self,user}` 四层叠加的 compounding 机制方向一致)

## 2. GCC 三元组作为映射语言（📘 来源：docx）

**定位声明（第三方审核 4.1）**：GCC 是 V2 的**对外沟通和分析辅助词汇**，**不是主骨架**。V2 的主骨架仍是 V1 工件八件套（见 §1.1）。下文凡涉及 GCC 之处，都可以通过 §2.2 翻译表回落到 V1 原语。

### 2.1 三元组定义（沿用 docx）

| 维度 | 问题 | 对应技术（docx 原表） |
|---|---|---|
| `Goal` 目标 | 人更快地与 Agent 交互完成目标 | Skill、交互设计、意图识别 |
| `Context` 上下文 | 优化 Agent 可用的上下文 | Harness Engineering、Memory、RAG |
| `Constraints` 约束 | 控制 Agent 的执行边界 | Sandbox、CICD、Lint、SPEC |

### 2.2 V1 工件到 GCC 的翻译表

V2 **不重命名** V1 工件；下表只做语义映射，方便未来文档在 GCC 语境和 V1 语境之间来回引用。

| V1 工件 | GCC 映射 | 说明 |
|---|---|---|
| `contract.product_goal` / `intent` | Goal | 目标声明 |
| `contract.product_acceptance` | Goal × Constraints | 目标的可验收形式 |
| `contract.non_goals` | Goal | 目标的负集 |
| `context-build.sh` 产出 | Context | 每次派发时构造的 runtime 上下文 |
| `memory/task` / `memory/self` / `memory/user` / `memory/views` | Context（长期） | 跨会话知识 |
| `state.context_*` | Context（短期） | 任务级控制面 |
| `state.policy` / `.team` | Constraints | 多 agent 策略、锚点分配 |
| `AGENTS.md` | Constraints | 协作规则、Dangerous Ops、Self-Reference Ops |
| `hooks/*` | Constraints | 执行期护栏 |
| `Cross-Repo Execution` 四字段 | Constraints | 跨仓边界 |
| `Task Size Gradation` S0-S3 | Constraints | 任务粒度硬约束 |
| V2 新增 `memory/skills/*.md` | Context（可执行投影） | 结构化 know-how |

**关键判断**：V2 **不**把 Skill 放进 `Goal` 轴（docx 原图把 Skill 归 Goal）。理由见 §2.3。

### 2.3 V2 与 docx 在 Skill 定位上的差异

docx 把 Skill 归为 `Goal` 轴技术，理由是它服务于"人更快和 Agent 交互完成目标"。

V2 不这样划。V2 把 Skill 归为 `Context` 轴的**可执行化投影** —— 因为在 DD Hermes 的落地语义里，Skill 是一段被 Agent 在运行时按需加载的 markdown know-how，本质是 Context 的压缩形态，而不是新的目标交互入口。

这不是否定 docx，而是把 docx 的抽象词落到 DD Hermes 的具体工件上。同一个 Skill 文件在不同项目里承担不同作用，但它始终是 Context 的投影，不直接是目标本身。

## 3. 五条道的 V2 表述（📘 来源：docx；只作为分析语言，不是 V2 协议骨架）

**弱化声明（偏懒修 + 第三方审核 3.5 精神）**：五条道是 docx 的分析框架，"V2 想强化的点" 一列是 🔀 头脑风暴 / 映射假设，**不等于 V2 立案承诺**。若不存在对应的 `provable_need` 就只作为分析语言保留。特别要承认：

- **信任之道 / 启发之道**：V2 对应较弱。"信任之道"的 V2 强化其实是"V2 不新造 runtime 层"的**保守声明**；"启发之道"用 `progressive disclosure` 对应只是词面近似
- **决策之道**：对应的候选 `v2-decision-gate-spec` 已降权为 C 证据项（见 `01-方向矩阵.md §1.3`），不作为立案承诺
- **积累之道 / 减法之道**：V2 对应较稳（`memory/{task,self,user,views}` 已落地、`Self-Reference Ops` / `Task Size Gradation` 已落地）

| 道 | V1 已落地机制（📂 仓库真相） | V2 想强化的点（🔀 映射假设） |
|---|---|---|
| **信任之道**（知信而非盲信） | Product/Quality Anchor 协议层常驻；`discussion.policy == 3-explorer-then-execute` | V2 不新造"管理 agent"；继续靠协议席位而非长期在线子 agent |
| **启发之道**（Inspire Prompt） | `contract.product_goal` / `non_goals` 强调"品味与边界"而非实现步骤 | V2 的 `memory/skills/*.md` 通过 progressive disclosure 让品味标准按需加载，降低 prompt 噪音 |
| **决策之道**（SPEC 优先于 Harness） | `openspec/` + `workspace/contracts/` + `quality-gate.sh` + `thread-switch-gate` | V2 方向矩阵候选 `v2-decision-gate-spec`：`product_acceptance` 从文本 → 可执行断言 |
| **积累之道**（Knowledge Compound） | `memory/task` / `memory/self` / `memory/user` / `memory/views` 四层 | V2 新增 `memory/skills/` 作为 Compound 载体；候选 `v2-compound-loop-review` 把蒸馏从被动变主动 |
| **减法之道**（Thin Everything） | `Task Size Gradation` S0-S3；`Self-Reference Ops` 拦截 harness 自指漂移 | V2 保留 S0-S3 + Self-Reference Ops；不为 V2 而膨胀流程；V2 新增文档只放 `V2/` 顶层，不污染 V1 |

## 4. 五步闭环对齐当前仓库现状

docx 的五步闭环：`Context Scout → Co-spark → Multi-lens Lock → Guarded Flow → Compound Loop`

| 步骤 | V1 当前落地 | V2 空白 / 候选支柱 |
|---|---|---|
| Context Scout | **缺失**：立 task 前的 context 盘点仍靠聊天记忆；`successor-evidence-audit.sh` / `dd-hermes-successor-evidence-audit-v1` 已暴露这是真实需求 | 候选 `v2-context-scout` |
| Co-spark | **缺失**：`3-explorer` 是探索视角而非 N 并行重写实现 | 候选 `v2-co-spark`（opt-in，见 §8.1） |
| Multi-lens Lock | 已有 `3-explorer-then-execute` + Product/Quality Anchor 协议 + `T3/T4` 多 agent 策略 | V2 不改 |
| Guarded Flow | 已有 `quality-gate.sh` + `thread-switch-gate` + `check-artifact-schemas.sh` + `hooks/*` | V2 候选 `v2-decision-gate-spec` 强化（schema 化 + 可执行 acceptance） |
| Compound Loop | 部分：`memory-write.sh` 已存在，但主动蒸馏缺失；`openspec/archive/*` 19 条，`memory/self/*` 只 2 条（蒸馏率约 10%） | 候选 `v2-compound-loop-review` |

## 5. Agent 体工学三原则的 V2 落地（📘 来源：docx 6.4；📂 V1 真相对照）

docx 6.4 提出 Agent Experience 三原则：最小化摩擦、最大化信息密度、信任模型能力。

**诚实自评（偏懒修）**：V2 当前对三原则的**直接用户触点**只有一条（`v2-textbook-daily`，唯一直接用户收益）；其他触点**大多是 V1 已落地的机制**，或 V2 harness enabler 的**间接影响**。下表明确区分"V1 已有"、"V2 想加但仍是 harness enabler"、"V2 直接用户触点"。

| 原则 | V1 已有（📂） | V2 想加（多数 harness enabler 🔀） | V2 直接用户触点（🎯） |
|---|---|---|---|
| 最小化摩擦 | `./scripts/demo-entry.sh` 唯一入口；`quality-gate.sh` 统一收口 | `v2-context-scout` 做立案前 manifest（manifest 给 agent 用，非用户端） | 暂无（7 候选中没有直接改善用户端摩擦的 slice） |
| 最大化信息密度 | `state.json` 单文件覆盖控制面；`AGENTS.md` + `docs/` 做了信息压缩 | `v2-skill-stack` 的 progressive disclosure（Agent 端收益，非用户端）| 暂无 |
| 信任模型能力 | `contract` 写 `product_goal + acceptance + non_goals` 而非 implementation prompt；`Self-Reference Ops` 反向拦截膨胀 | 无新增 | `v2-textbook-daily` 让用户每天看到 agent 做了什么，不用 micro-review（间接信任建设） |

**结论**：三原则的真正 V2 直接落地点很窄，目前仅 `v2-textbook-daily` 一条。这也回证了 `01-方向矩阵.md §1 前言` 的分层观察：V2 第一轮候选以 harness enabler 为主，对"直接用户触点"的覆盖不足，需要继续补充真正的直接用户收益方向。

## 6. Skill 蒸馏的可能演化线（📘 docx 终章 + 🔀 猜测；偏懒修降温）

docx 终章主张：Skill 是起点不是终局；未来是"蒸馏人类专家知识"的更高抽象（例如 Garry Tan 13 年创投经验的 AI 套件 / 厨师的 Cooking Stack）。

**降温声明（偏懒修）**：下表只是**分析性猜测**，不是 V2 承诺的实施路径；`v2-skill-stack` 目前只是**实验性 know-how 库**（见 `01-方向矩阵.md §1.1`），是否成为稳定层、是否演化到更高抽象，都要由实验结果决定。"Stack" 一词已在方向矩阵收敛为"更稳定的 know-how 库"，避免与 GCC 并列成模糊新层级。

| 阶段 | 状态 | 落实程度 |
|---|---|---|
| `memory/task`（任务级沉淀，人手触发 memory-write） | 📂 V1 已在运行 | 稳定 |
| `memory/self`（跨任务教训，人手触发） | 📂 V1 已在运行但稀疏（2 条 vs 19 条归档，蒸馏率约 10%） | 稳定但稀疏 |
| `memory/skills/`（按需加载的 know-how） | 🔀 **未创建** —— `v2-skill-stack` 候选若立 slice，以**实验**为目标（手动或半自动蒸馏）先验证复用价值 | **实验性，未决是否成为稳定层** |
| 更高抽象的 know-how 库（领域 DSL / cooking stack 类） | 🔀 docx 远景，**V2 范围之外**；不讨论、不承诺 | 远景 / 分析框架 |

## 7. 离经叛道的四条开放命题

V2 允许**重新思考** V1 的以下预设，但每条都必须附反证理由，不为差异而差异。当前倾向是"保留 V1 默认，待 Codex APP 使用场景出现真实冲突再升级"。

### 7.1 "对外一个线程"是否仍然必要？

- **V1 立场**：默认单线程对外，内部多角色切换
- **可能的反证**：Codex APP 用户同时在多个项目里工作（XC-BaoXiaoAuto / XC-Park-AI / DD Hermes 本身），每个项目都要一个长期 Agent
- **V2 当前倾向**：保留 V1 默认；在方向矩阵登记 `v2-multi-codex-fleet` 候选，`self_ref_risk=high`，`deferred` 直到用户实际并行项目 ≥ 2 且出现 cross-project skill 复用需求

### 7.2 "memory 和 state 不能互相替代"是否在 Skill 时代仍严格？

- **V1 立场**：memory 长期知识，state 短期控制面；严格分离（见 `AGENTS.md → Behavior Rules`）
- **可能的反证**：Anthropic Skills 的"Everything is Skill"主张让 Skill 跨越两者
- **V2 当前倾向**：保留 V1 分离；Skill 作为 **Context 的可执行投影**（见 §2.3），不替代 memory/state 本身；新增 `memory/skills/` 是 memory 子目录而不是独立第四层

### 7.3 工件命名是否按 GCC 重命名？

- **V1 立场**：`contract / state / context / handoff / worktree / closeout / exploration / archive` 八件套已稳定，所有脚本、hooks、memory references 都绑这套名字
- **可能的反证**：GCC 顶层语言出现后，工件名可能引起词汇混乱
- **V2 当前倾向**：**不重命名**；只在本总纲 §2.2 提供翻译表；保证所有现有脚本、hooks、memory references 不被破坏

### 7.4 "harness 自身不做 runtime"是否需要破例？

- **V1 立场**：DD Hermes 不是新 runtime，不做长期在线 daemon（见 `README.md` "它不是什么"）
- **可能的反证**：Compound Loop 自动化（每次 closeout 主动蒸馏）可能需要事件驱动的 daemon
- **V2 当前倾向**：**先做脚本化**（例如 `scripts/compound-review.sh` 手动或 cron 触发），不做 daemon；留给 V3 远景

## 9. 与 V1 协议的接口边界

V2 是**平行扩展层**，不是协议覆写层。

### 9.1 V2 不动 V1 的任何文件

不动：`AGENTS.md` / `README.md` / `CLAUDE.md` / `docs/*` / `指挥文档/*` / `hooks/*` / `scripts/*` / `tests/*` / `workspace/*` / `memory/*` / `openspec/*`。

若 V2 支柱立 slice 时需要修改 V1 文件，走正常的 V1 S2/S3 slice 流程，不搞"V2 批量改 V1"。

### 9.2 冲突以 V1 为准
