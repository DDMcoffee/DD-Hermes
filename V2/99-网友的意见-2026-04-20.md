# DD Hermes V2 综合审核意见

审核对象：
- `V2/README.md`
- `V2/00-哲学总纲.md`
- `V2/01-方向矩阵.md`

## 0. 基线事实

本轮综合判断延续两份原审核的共同前提：

- `./scripts/demo-entry.sh` 当前显示：`一期已经达到完整的 phase done`
- 当前 `active mainline`：`暂无`
- `指挥文档/06-一期PhaseDone审计.md` 与入口脚本口径一致

因此，V2 当前更适合被定义为：在 V1 稳定基线之上的下一轮产品方向与哲学整理，而不是 V1 的替代方案、迁移方案或已排好执行顺序的 roadmap。

## 1. 综合结论

结论仍然是：`建议先修文档再推进`。

这不是因为 V2 的方向来源本身错误，也不是因为生成过程违背了你的输入，而是因为最终文档里有四层内容还没有被充分分层表达：

1. 用户明确输入
2. 仓库现有真相
3. 外部参考筛选后吸收
4. 头脑风暴与多 Explorer 讨论结果

现在的主要问题，不是“方向错”，而是“边界、来源、优先级和工件落点还没有完全写稳”。

## 2. 这轮应明确判定为“做对了”的部分

- 先检查 V1 是否 `phase done`  
  这一步是对的，而且有本仓事实源支撑。

- 使用独立 `V2/` 目录承接大版本  
  这不是漂移，而是对“不要覆盖 V1、另起大版本目录”的直接落实，也是当前最稳的边界设计之一。

- 使用多 Explorer / 头脑风暴  
  这符合当时“Explorer 视角不够，建议多几个”的明确输入。问题不在于使用了头脑风暴，而在于最终文档是否把这层内部分析痕迹暴露得过重。

- 把 `XC-Park-AI` 保持为候选池，而不是直接绑定依赖  
  这一步是克制的，符合“先判断需不需要，再考虑使用”的原始要求。

- `docx` 没有被全盘照搬  
  当前文档已经有初步筛选与拒绝动作，方向是对的；问题在于筛选痕迹还不够显性。

## 3. 必改项

### 3.1 统一 `skills` 的唯一物理落点

涉及位置：
- `README.md:40-48`
- `00-哲学总纲.md:42`
- `00-哲学总纲.md:59-61`
- `00-哲学总纲.md:111`
- `00-哲学总纲.md:128`
- `00-哲学总纲.md:161`
- `00-哲学总纲.md:179-180`
- `01-方向矩阵.md:18-33`

问题：
- 一处写未来落在 `memory/skills/`
- 一处又举例成 `V2/skills/*.md`
- 同时又说 `memory/skills/` 当前不存在

建议：
- 先明确一个 canonical 物理目录
- 另一处只保留概念映射，不再作为真实落点出现
- 如果真实 slice 仍走 V1 路径，就不要在 `V2/` 再造第二套工件树

理由：
这是协议边界问题，不是措辞问题。路径不统一，后续任何 `v2-skill-stack` 都会落到错误目录。

不改风险：
- 形成两套 `skills` 树
- 稀释现有 `memory/state/context` 边界
- 让读者误以为 V2 已经引入新工件层

### 3.2 把 `01-方向矩阵` 从“像排期”收回到“候选池”

涉及位置：
- `01-方向矩阵.md:16-33`
- `01-方向矩阵.md:35-136`
- `01-方向矩阵.md:151-163`
- `01-方向矩阵.md:206-237`

问题：
- 虽然文档声明自己是候选池，不是 PRD，不是立案承诺
- 但正文大量出现 `首批 A`、`首批 B`、`二批`、`三批`、`可立即立 S2 slice`、`first_slice_shape`、`target_repo` 首选等接近执行排期的表达
- 结合生成过程补充，这些内容确实来自多 Explorer 头脑风暴，但这并不改变它在最终文档中“过像路线图”的问题

建议：
- 把批次语言降格为“证据强度排序 / 讨论顺序 / 观察优先级”
- 去掉近执行指令语气
- 评分与批次若保留，尽量后移到附录

理由：
用户要求“好的 idea 需要头脑风暴”，不等于最终文档要把 brainstorming 的内部痕迹直接外显成产品表面。

不改风险：
- 候选条目会被直接当成待执行任务
- `provable_need` 的再验证被跳过
- 方向矩阵滑成默认 roadmap

### 3.3 把“直接用户收益”与“harness enabler”明确拆层

涉及位置：
- `README.md:7-10`
- `00-哲学总纲.md:56-75`
- `01-方向矩阵.md:151-163`
- `01-方向矩阵.md:208-230`

问题：
V2 北极星写的是“更好使用 Codex APP、给 Codex 套上马具、沉淀用户知识/经验”，但后半段若干方向已经更像 harness / 治理 / 编排层优化，并被放到了接近同等优先级的位置。

建议：
- 候选支柱分成 `直接用户收益` 与 `harness enabler`
- 早批优先前者
- 后者附更高证据门槛，默认后移

理由：
V2 不是为了把控制面再做一轮漂亮自我优化，而是为了让真实使用体验和知识沉淀更顺滑。

不改风险：
- V2 继续滑向“自我优化的 harness roadmap”
- 产出很多控制面资产，但用户体感收益不明显

### 3.4 把“用户明确输入”和“筛选后吸收”显性区分开

涉及位置：
- `README.md:7-10`
- `00-哲学总纲.md:13-24`
- `00-哲学总纲.md:142-161`

问题：
最终三份文档没有足够清楚地告诉读者：哪些是用户原始要求，哪些是仓库事实，哪些是从 docx / GitHub 参考里筛选吸收来的。

建议：
- 在最终文档里显性区分：
  - 用户明确要求
  - 仓库现有真相
  - 外部参考筛选后吸收

理由：
这样读者才知道哪些是 DD Hermes V2 的原生约束，哪些只是参考框架。

不改风险：
- 文档显得像统一大理论
- 后续维护者无法判断哪些能动、哪些不能动

### 3.5 `docx` 的筛选痕迹要再显性一点

涉及位置：
- `00-哲学总纲.md:142-161`
- `01-方向矩阵.md:224-230`

问题：
目前方向是对的，但仍不够“让人一眼看出你没有拿来就用”。尤其结合生成过程补充，可以看出 docx 在早期被定性得偏快。

建议：
- 更明确写出：
  - 哪些只是映射语言
  - 哪些被明确拒绝
  - 哪些只保留为远景或分析框架

理由：
用户的原始要求是“先判断有无价值，再参考”，最终文档需要更显性地证明这一步真的发生了。

不改风险：
- 读者会把 V2 理解成 docx 的本地翻版
- 而不是 DD Hermes 基于自身事实做出的选择

### 3.6 `v2-decision-gate-spec` 仍应降权

涉及位置：
- `01-方向矩阵.md:53-73`
- `01-方向矩阵.md:159-160`

问题：
当前证据只直接证明 `schema check / gate` 覆盖率有缺口，并不直接证明 `contract` 字段必须 schema 化；这条方向更像“值得研究”，还不够支撑当前优先级。

建议：
- 保留方向
- 但降成 `待补证 / 实验项`
- 等出现明确的 `contract field drift` 或直接用户痛点，再重新升级

理由：
不能把治理层猜想提前塑造成明确下一步。

不改风险：
- 提前投入 JSON Schema / DSL 工程
- 把覆盖率问题扩大成一整套额外规范体系

### 3.7 `v2-skill-stack` 要收窄成实验性 know-how 库

涉及位置：
- `00-哲学总纲.md:97-112`
- `01-方向矩阵.md:16-33`

问题：
这条方向现在已经有来源、有证据、有用户目标支撑，但当前表述过于成熟，太快写成了“新层级”，而不是“小切口验证”。

建议：
- 不否掉这个方向
- 明确写成“实验性 know-how 库”
- 先验证复用价值，再决定是否升级成稳定层

理由：
“有蒸馏需求”不等于“现在就适合引入新层级”。

不改风险：
- 维护者默认 `skills` 已是 V2 正式层
- 过早固化一个还未验证的结构

## 4. 可选优化

## 5. 本轮确认的稳定点

- `README.md` 已较清楚表达：V2 是平行扩展层，不是协议覆写层
- `00-哲学总纲.md` 已明确：不重命名 V1 工件、冲突以 V1 为准、真实 slice 仍落在 V1 路径
- `00-哲学总纲.md` 与 `01-方向矩阵.md` 对 `Skill -> Context` 的归类是一致的
- 当前文档已明确“不做 daemon / 不做 runtime”这一谨慎边界
- `README` 与 `01-方向矩阵` 的主边界上，已经写出“当前尚未立 slice，这里只是候选池”

## 6. 对 `Agentic之道 V2.docx` 的综合判断

### 6.1 可吸收

- GCC 作为对外沟通与映射语言
- `Context` 是核心资产这一判断
- `Agent 体工学` 作为用户摩擦检查视角

### 6.2 谨慎借鉴

- 五条道、五步闭环
- `Skill 不是终局`
- `Knowledge Stack`
- `v2-skill-stack`
- `v2-decision-gate-spec`
- `v2-co-spark`

这些更适合作为分析框架、远景或映射语言，不适合直接固化成 V2 的协议骨架或近期开工路线。

### 6.3 明确不吸收

- `大力出奇迹`
- 默认烧 token
- 默认多 agent 反复重写
- `Everything is Skill`
- 默认 fleet / daemon runtime / 对外多线程常态化

## 7. GitHub 参考项目的综合取舍

### 最有价值

- `GitHub Spec Kit`
  - 适合参考 `spec-driven`、`contract-first`、可执行验收门槛
  - 不应照搬命令流、目录仪式和 CLI 外形

- `Fission-AI OpenSpec`
  - 适合参考 `artifact-guided workflow`
  - 不应把 V2 直接做成另一个 OpenSpec 壳

- `Mem0`
  - 适合参考长期记忆层的层次化与持续演进
  - 不应把“可检索”误当成“已记住”

### 有参考价值，但需克制

- `Agent-Skills-for-Context-Engineering`
  - 适合参考 `skills` 作为按需加载上下文资产
  - 不应因为它 skill 体系完整，就默认 DD Hermes 当前也该完整引入

- `MetaGPT`
  - 适合参考多角色协作和协议化团队分工思想
  - 更适合作为上界样本，而不是当前 V2 落地模板

- `Acontext`
  - 适合参考 `memory -> skill file` 这种显式化做法
  - 不应默认走自动化升级流，仍需小证据闭环

### 不纳入当前主方向

- `A2A / MCP-first` 的协议崇拜
- `群聊即系统` 的多 agent 炒法

## 8. 建议保留与建议降权

### 建议保留

- `v2-skill-stack`，但必须收窄
- `v2-context-scout`
- `v2-textbook-daily`，但先只做最小 daily summary
- `v2-compound-loop-review`，作为 `v2-skill-stack` 之后的条件保留项

### 建议降权或后移

- `v2-decision-gate-spec`
- `v2-co-spark`
- `v2-multi-codex-fleet`

## 9. 建议的修文顺序

1. 统一 `skills` 的唯一物理落点
2. 把 `01-方向矩阵` 从“排期语气”收回到“候选池语气”
3. 把 `直接用户收益` 与 `harness enabler` 拆层
4. 把“用户明确输入 / 仓库真相 / 外部参考筛选后吸收”三层显性区分
5. 把 `docx` 的吸收与拒绝痕迹再显性化
6. 降权 `v2-decision-gate-spec`
7. 收窄 `v2-skill-stack`
8. 最后再处理 GCC / 五条道 / 五步闭环作为主骨架的问题
