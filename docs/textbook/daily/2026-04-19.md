## Confirmed Conclusions

- “预览优先”能把 OCR 的不确定性留给用户确认：先把候选行拿出来，再决定哪些要落库，比“上传就自动创建”更稳。
- “不改 schema 的策略”在这一类切片里是可行的：list-only 字段（往返、AM/PM 等）可以先落到 `notes`，把建模压力推迟到真的被需求逼出来那一刻。
- “清理脏改动”的正确方式是对照落地基线逐条审判：留下来的每一行都要回答“它现在还解决真实 gap 吗？”否则 `git status` 会变成噪音。
- “并发 build 失败”不一定是功能回归：当错误像缓存/竞态时，先做串行重跑与 fresh build 的证据比猜原因更值钱。

## Reusable Patterns

- 跨仓切片（S1/S2）统一动作顺序：`contract -> handoff -> state -> closeout -> (land) -> verify -> archive`，并把 `target_repo_ref` 写死，避免“验收时不知你到底跑的是哪版代码”。
- UI 流程做成可复用的“五步模板”：入口（页面按钮）→ 上传（明确 kind）→ 轮询（可取消/超时）→ 预览（可选/可删）→ 确认（批量创建 + 错误提示）。
- Degraded 不是羞耻，是状态：只要写清“没有独立质量位”并把验证命令落盘，就比“假装有人看过”更可靠。
