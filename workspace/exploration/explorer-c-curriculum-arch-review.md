## 2. Memory Read / Retrieval

### 2.1 memory-read.sh

Retrieval is **lexical word-bag matching**:
1. Tokenize task_context by splitting on whitespace, /, -
2. For each memory card, score = count of tokens appearing in (content + scope)
3. Constraint type gets +100 bonus score
4. Sort by -score, take top N (default 10)
5. Also scans journal for conflict events related to selected memory IDs

### 2.2 Issues

- **No semantic search** - Word-bag matching will degrade as memory count grows. A card about "data visualization" won't match a query about "chart rendering" unless exact tokens overlap.
- **No recency weighting** - A card from 6 months ago scores the same as one from today if tokens match equally.
- **No confidence weighting** - A card with confidence 0.1 ranks the same as one with 1.0 if token scores match.
- **Hard limit, no relevance threshold** - Returns top N regardless of score quality. Could return irrelevant cards if few tokens match.

### 2.3 Mitigation

For the current scale (5 cards), word-bag is sufficient. The risk materializes at ~50+ cards or when topics diversify beyond the current Codex/Hermes domain.

## 3. Journal & Provenance

### 3.1 Event Schema

Both memory-write.sh and memory-manage.sh write journal events with:
```
event_id, memory_id, op, actor, timestamp, before, after,
reason, source_ref, scope, confidence_delta, result
```

This is a solid provenance schema. The before/after snapshots enable full audit trail.

### 3.2 Issues

- **Journal directory is empty in main workspace** - memory/journal/ has 0 items. All journal entries were created in smoke test tmpdir and not persisted. This means the real workspace has zero provenance history.
- **No journal compaction** - Append-only JSONL with no rotation or compaction. Long-running projects could accumulate large files.
- **Journal is not consumed downstream** - Only memory-read.sh scans journal for conflicts. No other script reads journal for analytics, decay scheduling, or knowledge evolution tracking.
